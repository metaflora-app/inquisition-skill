# Metaflora Inquisition

> Скилл для проверки утверждений под давлением: источник, логика, контекст, точка разлома, финальная булла.

![скетчноут: конвейер инквизиции](assets/inquisition-pipeline-sketchnote.png)

## Предисловие

В агентной работе факты часто приходят в красивой упаковке.

Цифра выглядит уверенно. Цитата звучит знакомо. Сравнение моделей опирается на benchmark, который кто-то уже пересказал. Конкурент пишет громкое заявление в лендинге. Другой AI выдаёт ответ с аккуратной структурой, ссылками и тоном человека, который якобы всё проверил.

Проблема начинается в момент, когда уверенный тон принимают за доказательство.

`metaflora-inquisition` создан для таких моментов. Его задача — заставить утверждение пройти процедуру проверки. Сначала установить происхождение. Затем разобрать источники, логику и контекст. Потом решить, хватает ли данных для вердикта. Если данных мало или выводы конфликтуют, скилл назначает точечное давление. В финале он выдаёт буллу: структурированный документ с вердиктом, оговорками, исправленной формулировкой и уровнем доверия к источникам.

Название театральное, процесс практический.

Внутри нет декоративного roleplay ради roleplay. Средневековый язык нужен как рабочая рамка: он делает проверку жёсткой, последовательной и запоминающейся. Агент перестаёт мягко соглашаться с тезисом. Он начинает вести разбирательство.

## Коротко

**Metaflora Inquisition** — переносимый skill для AI-агентов, который превращает запросы вроде `это правда?`, `перепроверь`, `fact-check`, `/inquisition` и `допроси это` в полный процесс верификации.

Он подходит для:

- статистики;
- цитат;
- исследований;
- прогнозов;
- сравнений моделей;
- продуктовых claims;
- заявлений конкурентов;
- выводов других AI;
- планов, основанных на данных;
- постов, презентаций и лендингов, где фактическая ошибка стоит дорого.

Главный результат работы — `BULLA INQUISITIONIS`, финальный документ с одним вердиктом.

## Зачем появился этот skill

AI-агент легко превращает проверку в поиск удобной ссылки. Особенно если пользователь формулирует задачу мягко: `убедись`, `проверь`, `точно?`.

Такой режим слаб для нескольких типов задач.

Во-первых, вторичные источники часто ссылаются друг на друга. Маркетинговый блог пересказывает другой блог, тот ссылается на "данные Meta", конкретный отчёт отсутствует. Внешне всё выглядит убедительно, внутри пусто.

Во-вторых, источник может существовать, но утверждение использует его слишком широко. Исследование проверяло одну аудиторию, а вывод применяют ко всем. Benchmark покрывал один тип задач, а в посте он превращается в доказательство общего превосходства модели.

В-третьих, контекст стареет. Цифра могла быть честной в 2021 году, затем рынок изменился, механика платформы обновилась, аудитория привыкла к формату, и прежний вывод стал слабым.

В-четвёртых, другой AI может уверенно собрать смесь из реальных терминов, устаревших фактов и несуществующих ссылок. Обычная аккуратная структура ответа усиливает доверие к тексту, хотя сама структура ничего не доказывает.

`metaflora-inquisition` решает эту проблему процедурой. Агент получает маршрут проверки с фиксированными стадиями, независимыми ролями и финальным документом.

## Главная идея

Утверждение должно доказать пригодность.

Скилл держит нейтральную позицию по отношению к пользователю, автору источника, конкуренту, benchmark-команде и предыдущему AI. Он разбирает тезис как объект проверки.

Каждая стадия добавляет свой слой:

- происхождение;
- источник;
- логика;
- контекст;
- точка разлома;
- итоговый вердикт.

Если тезис выдерживает проверку, он получает `ABSOLVO`.

Если тезис держится только с важными оговорками, он получает `ABSOLVO CUM NOTA`.

Если в тезисе есть исправимое ядро, он получает `PENITENTIA`.

Если тезис противоречит проверенным данным, он получает `ANATHEMA`.

Если данных недостаточно, он получает `IGNORAMUS`.

## Архитектура процесса

```text
ADMONITIO
  ↓
INTERROGATIO
  ├─ Fontium
  ├─ Rationis
  └─ Contextus
  ↓
DECISIO TRIBUNALIS
  ↓
TORTURA при необходимости
  ├─ Advocatus Diaboli
  └─ Advocatus Fidei
  ↓
BULLA INQUISITIONIS
```

## Стадия I: ADMONITIO

`ADMONITIO` — стадия происхождения.

На этом шаге агент ищет:

- кто автор утверждения;
- где впервые опубликован тезис;
- есть ли живой URL;
- какие данные лежат в основе;
- является ли источник первичным;
- кто получает выгоду от распространения тезиса.

Стадия мягкая по форме и строгая по назначению. Её цель — выяснить, существует ли опора у утверждения.

Возможные статусы:

| Статус | Значение |
| --- | --- |
| `источник найден` | URL живой, данные совпадают с тезисом |
| `источник не установлен` | первоисточник найти не удалось |
| `предварительное опровержение` | прямое противоречие найдено уже на входе |

## Стадия II: INTERROGATIO

`INTERROGATIO` — ядро скилла. Здесь три инквизитора проверяют тезис независимо.

![скетчноут: три инквизитора](assets/three-inquisitors-sketchnote.png)

### Fontium

`Fontium` отвечает за источники.

Он проверяет:

- существует ли первоисточник физически;
- открывается ли URL;
- совпадает ли цитата или цифра с оригиналом;
- является ли источник первичным;
- когда данные обновлялись;
- есть ли конфликт интересов;
- насколько длинная цепочка пересказов.

Хороший вывод `Fontium` не рассуждает широко. Он фиксирует состояние источников.

Примеры уязвимостей:

- "40% указаны без первичного отчёта";
- "цитата найдена только во вторичном пересказе";
- "исследование опубликовано поставщиком продукта";
- "дата обновления данных неизвестна";
- "оригинальный документ недоступен".

### Rationis

`Rationis` отвечает за логику.

Он не ходит в интернет. Его поле — структура утверждения.

Он проверяет:

- противоречия внутри тезиса;
- подмену понятий;
- вывод шире данных;
- correlation/causation;
- circular reasoning;
- скрытые допущения;
- cherry-picking;
- неопределённые термины вроде "лучше", "эффективнее", "быстрее".

Примеры уязвимостей:

- "лучше" не определено метрикой;
- "рост охвата" смешан с "ростом продаж";
- "выборка" подана как общий рынок;
- "benchmark" используется как доказательство всех real-world задач;
- "корреляция" подана как причина.

### Contextus

`Contextus` отвечает за контекст.

Он проверяет:

- где тезис появился;
- на какую аудиторию распространялся;
- к какому периоду относился;
- какие нюансы исчезли при пересказе;
- применим ли исходный вывод к ситуации пользователя;
- кому выгодно упрощение контекста.

Примеры уязвимостей:

- "данные относятся к рынку США";
- "исследование касалось enterprise-команд";
- "цитата дана в разговоре о другой задаче";
- "вывод устарел после изменения API";
- "результат верен только для одной версии модели".

## Стадия III: DECISIO TRIBUNALIS

Трибунал получает выводы трёх инквизиторов и решает глубину проверки.

Его задача — выбрать маршрут:

- сразу к булле;
- через `TORTURA`.

`TORTURA` назначается при конфликте выводов, серьёзной неопределённости, прямом противоречии или конкретной статистике без установленного источника.

Если выводы уже сходятся, трибунал может отправить дело сразу к булле.

## Стадия IV: TORTURA

`TORTURA` — точечное давление по одной выбранной точке.

Трибунал обязан назвать конкретный разлом. Пример хорошей точки:

```text
Проверить, существует ли отчёт Meta, где именно Reels с субтитрами получают +40% охвата, и показывает ли он причинный эффект субтитров.
```

Пример плохой точки:

```text
Проверить источники ещё раз.
```

На `TORTURA` работают два адвоката:

- `Advocatus Diaboli` атакует тезис;
- `Advocatus Fidei` защищает тезис.

Оба работают только по назначенной точке. После столкновения трибунал решает, кто победил в этой точке.

## Стадия V: BULLA INQUISITIONIS

`BULLA` — финальный документ.

В ней фиксируется:

- предмет разбирательства;
- статус `ADMONITIO`;
- выводы `Fontium`, `Rationis`, `Contextus`;
- проводилась ли `TORTURA`;
- финальный вердикт;
- оговорки;
- исправленная версия при необходимости;
- проверенные источники;
- уровень доверия к источникам.

Булла нужна для решений. Её можно вставить в документ, PRD, research note, пост, слайд, комментарий к задаче или внутренний decision log.

## Вердикты

![скетчноут: вердикты](assets/verdicts-sketchnote.png)

| Вердикт | Когда использовать | Что получает пользователь |
| --- | --- | --- |
| `ABSOLVO` | источник живой, логика держится, контекст сохранён | подтверждение пригодности тезиса |
| `ABSOLVO CUM NOTA` | тезис верен с важными ограничениями | список оговорок и границы применимости |
| `PENITENTIA` | ядро тезиса пригодно, формулировка требует правки | исправленную версию |
| `ANATHEMA` | тезис противоречит проверенным данным | корректную замену и причину отбраковки |
| `IGNORAMUS` | данных для проверки недостаточно | честную фиксацию неопределённости |

## Когда запускать

Используй skill, когда видишь такие фразы:

```text
/inquisition
инквизиция
допроси это
проверь под давлением
это правда?
перепроверь
я не уверен в этом
fact-check
проверь факт
убедись
```

Также запускай его для текстов такого типа:

- "по данным исследования...";
- "все знают, что...";
- "модель X лучше модели Y";
- "этот формат даёт +40%";
- "конкурент утверждает...";
- "AI сказал, что...";
- "benchmark доказывает...";
- "к 2027 году рынок вырастет...";
- "пользователи предпочитают...";
- "это подтверждает исследование".

## Когда сначала нужно сузить тезис

Иногда пользователь приносит слишком широкий текст.

Примеры:

```text
AI изменит образование.
Этот рынок будет огромным.
Наш продукт лучше конкурентов.
Локальные модели победят облачные.
```

В таких случаях сначала выдели атомарные утверждения:

```text
Нашёл 4 проверяемых утверждения:
1. рынок X вырастет до Y к 2028 году;
2. продукт A дешевле продукта B в enterprise-тарифе;
3. модель C быстрее модели D на задаче E;
4. пользователи сегмента S предпочитают формат F.
Проверяю каждое отдельно.
```

- исходное утверждение без переписывания;
- статус происхождения;
- вывод источников;
- вывод логики;
- вывод контекста;
- решение трибунала;
- `TORTURA`, если назначена;
- финальный вердикт;

## Пример: статистика

Запрос:

```text
инквизиция: Reels с субтитрами получают на 40% больше охвата
```

Что будет проверяться:

- откуда взялась цифра 40%;
- существует ли первичный отчёт;
- кто публиковал тезис;
- совпадает ли формулировка с источником;
- доказана ли причинность;
- применима ли цифра к текущему периоду;
- есть ли разница между охватом, удержанием, досмотром и продажами.

Возможный итог:

```text
ВЕРДИКТ: PENITENTIA

ИСПРАВЛЕННАЯ ВЕРСИЯ:
Субтитры могут улучшать удержание и досмотр в ряде контекстов, но конкретная
цифра +40% требует первичного источника, периода, рынка и определения метрики.
```

## Пример: сравнение моделей

Запрос:

```text
fact-check: Claude лучше GPT-4o для кодинга
```

Что будет проверяться:

- какая версия Claude;
- какая версия GPT-4o;
- какая дата сравнения;
- какие задачи входят в "кодинг";
- какой benchmark используется;
- кто проводил оценку;
- есть ли независимые данные;
- переносится ли benchmark на реальные проекты.

Возможный итог:

```text
ВЕРДИКТ: ABSOLVO CUM NOTA

ОГОВОРКИ:
Преимущество зависит от версии модели, типа задач, даты проверки, benchmark
методологии и стека разработки.
```

## Пример: прогноз

Запрос:

```text
инквизиция: рынок AI-агентов вырастет до 50 млрд долларов к 2030 году
```

Для прогнозов скилл проверяет обоснованность:

- кто сделал прогноз;
- какая методология;
- какой базовый год;
- какой CAGR;
- что входит в рынок;
- какие сценарии учитывались;
- какие неопределённости могут сломать прогноз.

В булле появятся:

- горизонт актуальности;
- ключевые неопределённости;
- уровень доверия к источникам.

## Закрытые данные

Если тезис опирается на внутреннюю аналитику, `Fontium` фиксирует ограничение:

```text
Fontium: верификация источников невозможна без доступа к закрытым данным.
```

После этого `Rationis` и `Contextus` продолжают работу:

- проверяют логику вывода;
- ищут скрытые допущения;
- проверяют применимость;
- предлагают, какие данные нужны для уверенного решения.

## Веб-поиск

`ADMONITIO`, `Fontium`, `Contextus`, `Diaboli` и `Fidei` используют веб-поиск.

`Rationis` работает по тексту утверждения.

Если веб-поиск недоступен, булла должна содержать пометку:

```text
DEGRADED: веб-поиск недоступен, проверка источников ограничена.
```

## Практические правила для агента

1. Сначала выдели атомарное утверждение.
2. Не переписывай предмет разбирательства в булле.
3. На `ADMONITIO` ищи происхождение, без финального вердикта.
4. На `INTERROGATIO` держи роли независимыми.
5. `Rationis` не использует веб-поиск.
6. Каждый инквизитор называет одну уязвимость.
7. Трибунал выбирает глубину проверки.
8. `TORTURA` бьёт в одну точку.
9. Булла содержит один вердикт.
10. При `PENITENTIA` и `ANATHEMA` добавь исправленную версию.

## Что считается хорошей уязвимостью

Хорошая уязвимость конкретна:

```text
Цифра 40% встречается во вторичных маркетинговых статьях, первичный отчёт не найден.
```

Слабая уязвимость расплывчата:

```text
Нужно ещё проверить источники.
```

Хорошая уязвимость помогает трибуналу назначить точку удара.

## Что считать первичным источником

Обычно первичными считаются:

- официальный отчёт;
- научная статья;
- dataset;
- документация продукта;
- пресс-релиз компании о собственном продукте;
- нормативный документ;
- финансовая отчётность;
- исходная публикация автора цитаты;
- raw benchmark results.

Вторичные источники:

- блог о чужом отчёте;
- новость о пресс-релизе;
- пост с пересказом исследования;
- агрегатор;
- AI-ответ;
- презентация без ссылок на данные.

## Уровень доверия к источникам

| Уровень | Значение |
| --- | --- |
| `HIGH` | первичный источник найден и жив |
| `MEDIUM` | вторичные источники найдены, первичный ограничен |
| `LOW` | есть только косвенные данные |
| `NONE` | источники отсутствуют |

## FAQ

### Можно ли использовать skill без интернета?

Можно, но проверка источников будет ограниченной. Булла должна явно сказать, что веб-поиск недоступен.

### Можно ли проверять несколько утверждений сразу?

Можно. Сначала декомпозируй текст на атомарные утверждения, затем запусти процесс для каждого.

### Можно ли проверять прогнозы?

Можно. Для прогнозов проверяется обоснованность, горизонт актуальности и ключевые неопределённости.

### Можно ли проверять утверждения другого AI?

Да. Это один из главных сценариев.

### Можно ли использовать skill для ревью research-документов?

Да. Выдели claims, затем проверь каждый claim отдельно.

### Можно ли использовать результат в публичной статье?

Да, если булла содержит проверенные источники и уровень доверия. Для публичного текста лучше сохранять URL и дату проверки.
